## Что важно при ручной проверке UI

- Не использовать фильтры `Бренд`, `Товар`, `Своя товарная категория`: сейчас в данных эти поля пустые.
- Поле `Категория` в текущем UI фактически работает как быстрый preset проблемы, а не как товарная категория.
- `top_products_by_problem` сейчас непоказателен: все строки сведутся к `товар без названия`.
- Для RAG не сужать редкий смысловой запрос неподходящим label-фильтром. Например, `хрупкий товар` + `Несоответствие карточке товара` почти гарантированно даст пусто: в данных по `хрупк%` только 2 отзыва с этим label.

## Хорошие смысловые запросы для RAG

Эти формулировки соответствуют реальным фразам в данных:

- `хрупкий товар разбился при доставке`
- `сломался через день`
- `пластик хрупкий отвалилось крепление`
- `маломерит на размер`
- `плохая упаковка товар пришел помятый`
- `цвет не соответствует фото`
- `вернуть деньги за бракованный товар`

Проверенный RAG-smoke:

- Шаблон: `RAG: похожие отзывы`
- Период: `01.09.2025..30.11.2025`
- Проблема: `Проблема с качеством товара`
- Смысловой запрос: `хрупкий товар разбился`
- Результат: 3 релевантных примера без warnings; тексты вида `Товар приехал сломанный`,
  `Товар пришёл сломанный, видимо при транспортировки`, `Товар пришёл сломанный`.

Для RAG сначала лучше не включать ограничение проблемами. Если нужен label-фильтр:

- `хрупкий товар` лучше сочетать с `Проблема с качеством товара` или `Проблема с комплектацией / упаковкой`;
- `маломерит` сочетать с `Проблема с размером / посадкой`;
- `цвет не соответствует фото` сочетать с `Несоответствие карточке товара`;
- `вернуть деньги` сочетать с `Проблема с возвратом`.

## Наблюдения по качеству данных

- В данных много мульти-label: `32569` отзывов с двумя labels, `1918` с тремя и более.
- `18201` отзыв одновременно имеют `Положительный / нейтральный отзыв` и хотя бы одну проблему. Это нормально для mixed sentiment, но усложняет метрику positive vs problem.
- Товарных атрибутов нет: нельзя честно проверять `top_products_by_problem`, brand/category/product filters и товарные срезы.
- Медианная длина отзыва `64` символа, p95 `266`, около `1259` отзывов короче 10 символов.
- Частотные keyword-якоря: `качество` 27606, `размер` 25688, `цвет` 13175, `упаковка` 13114, `доставка` 5494, `запах` 5136, `маломер` 3691, `возврат` 3521.

## Улучшения пайплайна

1. Добавить enrichment товарных атрибутов.
   Сейчас нет `product_id/product_name/category/brand/rating`, поэтому часть продуктовых сценариев выглядит сломанной, хотя backend работает.

2. Разделить sentiment и problem labels.
   `Положительный / нейтральный отзыв` лучше хранить как отдельное поле `sentiment`, а problem labels оставить отдельной multi-label таксономией. Это сделает `positive_vs_problem` интерпретируемым.

3. Добавить `primary_label`.
   Для UI и графиков полезно иметь основной класс проблемы, а multi-label оставить для cooccurrence и детального анализа.

4. Запустить кластеризацию внутри крупных labels.
   Приоритет: `Проблема с качеством товара`, `Проблема с размером / посадкой`, `Проблема с комплектацией / упаковкой`. Там много разных подпричин: сломалось, ткань/материал, хрупкость, маломерит/большемерит, упаковка/помято/неполный комплект.

5. Сформировать подпроблемы semi-supervised способом.
   Сначала HDBSCAN/KMeans по embeddings внутри label, потом LLM-названия кластеров, потом ручная валидация топ-кластеров.

6. Добавить keyword/topic слой рядом с labels.
   Практичные словари: `маломерит`, `большемерит`, `слом`, `хрупк`, `помят`, `упаков`, `цвет`, `запах`, `возврат`, `деньг`. Это даст быстрые фильтры и объяснимые drill-down.

7. Для RAG сохранять в Qdrant дополнительные payload-поля.
   Полезно добавить `primary_label`, `sentiment`, `topic_cluster_id`, `topic_cluster_name`, `text_len`, `has_photo`/`rating`, если появятся.

8. Добавить evaluation set для RAG.
   Нужен набор из 30-50 проверочных смысловых запросов с ожидаемыми labels/примерными отзывами, чтобы не проверять вслепую после каждой перезагрузки.

9. Улучшить UI guardrails.
   Скрывать `top_products_by_problem` и product/category/brand filters, если facets пустые. Для RAG показывать предупреждение, когда выбран редкий label + узкий смысловой запрос.
