# Улучшение LLM/RAG-аналитики отзывов

## Что уже заложено

Текущий online-пайплайн соответствует baseline-схеме:

```text
Chat / Template UI
  -> ParsedQuery
  -> ToolRouter
  -> PostgreSQL / Qdrant
  -> StructuredResult
  -> Template answer / LLM answer
```

## Методы, которые стоит развивать дальше

### 1. LLM parser для ParsedQuery

LLM должен превращать свободный вопрос пользователя в строгий `ParsedQuery`:

- `intent`;
- `filters`;
- `tools`;
- `answer_mode`;
- `semantic_query`.

Важно держать rule-based fallback, чтобы сервис не ломался без API-ключа или при невалидном JSON.

### 2. Hybrid retrieval

Для сложных вопросов полезно объединять:

- PostgreSQL: точные числа, доли, динамика, топы;
- Qdrant: похожие отзывы, смысловые примеры, формулировки жалоб.

Пример: “Что стало хуже у книг в сентябре?” сначала считает рост проблем в PostgreSQL, затем берет Qdrant-примеры по выросшим labels.

Важно: слова вроде “книги”, “обложки”, “страницы” не всегда являются точным значением колонки `category`. В чате их лучше трактовать как semantic scope:

```text
semantic_query = исходный вопрос пользователя
tools = ["postgres", "qdrant"]
filters.category = null
```

Дальше Qdrant ищет кандидатов, LLM валидирует релевантность кандидатов вопросу, а backend агрегирует labels валидных отзывов.

### 3. Диверсификация выдачи Qdrant

Если взять top-k напрямую, LLM часто получает много почти одинаковых отзывов. Лучше:

- забирать больше кандидатов, например `limit * 3`;
- группировать по `label`, `product_id`, `product_name`;
- выбирать round-robin из разных групп.

Это уже уменьшает повторяемость ответа без тяжелой кластеризации.

### 4. Кластеризация примеров

Для больших выборок можно добавить offline/online кластеризацию:

- HDBSCAN или k-means по embeddings;
- отдельные “темы” внутри одной проблемы;
- выдача 2-3 отзывов из каждого кластера.

Так ответ будет показывать разные причины: упаковка порвана, пришел мятый товар, не хватает деталей, плохая инструкция.

### 5. Reranking

После Qdrant top-k можно добавить reranker:

- cross-encoder;
- LLM rerank;
- простые бизнес-правила: свежие отзывы выше, низкий рейтинг выше, выбранный товар выше.

### 6. Evidence-grounded answer

Финальный LLM-ответ должен ссылаться только на `StructuredResult`:

- факты из PostgreSQL;
- примеры из Qdrant;
- warnings, если часть инструментов не сработала.

Нельзя просить LLM “угадать” причины без найденных отзывов.

### 7. Quality checks

Для контроля качества нужны проверки:

- ParsedQuery соответствует вопросу;
- labels не содержат positive label в проблемных сценариях;
- Qdrant examples релевантны semantic query;
- LLM answer не добавляет причин, которых нет в examples.
