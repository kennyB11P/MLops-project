# Архитектура приложения для анализа отзывов

## 1. Главный принцип

В сервисе есть два пользовательских входа:

```text
1. Шаблонная аналитика
2. Чат с LLM
```

Но дальше backend должен работать одинаково.

```text
Шаблонный UI → ParsedQuery
Чат с LLM   → ParsedQuery
```

`ParsedQuery` — это единый внутренний объект, который описывает:

- источник запроса;
- намерение пользователя;
- фильтры;
- нужные инструменты;
- режим ответа.

## 2. Общая online-схема

```text
Пользователь
   ↓
Шаблонная форма / Чат
   ↓
ParsedQuery
   ↓
Tool Router
   ↓
PostgreSQL / Qdrant
   ↓
StructuredResult
   ↓
Template Answer / LLM Answer
   ↓
Ответ пользователю
```

## 3. Offline pipeline

Offline pipeline работает до пользовательского запроса.

```text
Новые отзывы
   ↓
Очистка / нормализация
   ↓
Классификатор
   ↓
Сохранение labels в PostgreSQL
   ↓
Построение эмбеддингов
   ↓
Сохранение векторов в Qdrant
```

Классификатор не должен запускаться на каждый пользовательский запрос. Он заранее размечает отзывы, а online-сервис уже использует готовые labels.

## 4. PostgreSQL

PostgreSQL нужен для точных структурных запросов:

- количество отзывов;
- доля проблемы;
- динамика по дням/неделям/месяцам;
- топ товаров;
- топ брендов;
- топ категорий;
- сравнение периодов;
- фильтрация по датам, товарам, категориям и брендам.

## 5. Qdrant

Qdrant нужен для смыслового поиска:

- найти похожие отзывы;
- показать примеры жалоб;
- найти отзывы по свободной формулировке;
- достать типовые формулировки внутри класса проблемы.

## 6. Answer mode

Есть два режима ответа:

```text
template — сухой результат: число, таблица, список, топ;
llm      — аналитический вывод, объяснение, рекомендации.
```

Кнопка “Добавить аналитический вывод” в шаблонном UI просто меняет:

```text
answer_mode: template → llm
```

## 7. Почему структура вынесена в `service/`

Текущий проект уже содержит исследовательские ноутбуки и документацию. Поэтому прикладной слой лучше добавить отдельно:

```text
service/backend/   # API, модели, роутеры, инструменты
service/frontend/  # типы и компоненты для сайта
service/scripts/   # команды запуска и проверки
service/tests/     # тесты контрактов
```

Так можно постепенно превращать исследовательский проект в продукт, не ломая существующие файлы.
